# /wj:explain — 소프트웨어 공학을 모르는 사람을 위한 해설

<HARD-RULE>
질문자는 소프트웨어 공학을 모른다고 가정한다.
"컴포넌트", "훅", "미들웨어", "인스턴스" 같은 단어조차 처음 듣는 사람이다.
모든 전문 용어는 반드시 일상 언어로 풀어야 한다.
</HARD-RULE>

## 청중 기준

- 코딩은 AI한테 시키고, 결과물이 뭔지 궁금한 사람
- "이 파일이 뭐 하는 거야?"가 자연스러운 질문인 사람
- 프로그래밍 수업을 들은 적 없는 사람
- "변수", "함수" 정도는 알 수도 있지만 확신할 수 없는 사람

## 응답 구조 (4단계 — 반드시 이 순서로)

### 1. 한 줄 요약 + 비유 (What)

- 질문 대상을 **한 문장**으로 — 전문 용어 없이
- 반드시 **일상 비유** 하나 붙인다 (음식점, 택배, 학교, 도서관 등)
- 기술 비유 금지 ("파이프라인", "레이어" 같은 것도 비유로 안 됨)

> 예: "이 파일은 손님 주문을 받아서 주방에 전달하는 웨이터예요."
> 예: "이건 도서관 카드 목록이에요 — 어떤 책이 어디 있는지 정리해둔 표."

### 2. 전체 그림에서 어디? (Where)

소프트웨어를 **음식점**에 비유해서 위치를 알려준다:

```
🖥️ 화면 (홀)        손님이 보는 메뉴판, 버튼, 입력창
   ↕️ 주문 전달
🚪 통로 (API)       주문서를 주방으로 넘기는 창구
   ↕️ 요리 지시
🧑‍🍳 주방 (서버)      실제 일을 처리하는 곳 — 계산, 판단, 가공
   ↕️ 재료 꺼내기
🗄️ 냉장고 (DB)      데이터를 보관하는 저장소
   ↕️ 외부 배달
🚚 배달 (외부 서비스)  결제사, 지도, 이메일 등 남의 서비스
```

- 질문 대상이 위 중 **어디에 해당하는지** 굵게 표시
- "이 파일은 **🧑‍🍳 주방** 쪽이에요" 식으로 한 줄 배치
- 바로 앞뒤와 뭘 주고받는지 한 줄로

### 3. 왜 이렇게 만들었는지 + 다른 방법은? (Why — 가장 중요)

**이 단계가 이 스킬의 존재 이유다.**
"왜?"를 설명할 때도 전문 용어 없이, 비유의 연장선에서 설명한다.

반드시 포함:

1. **이렇게 만든 이유** — 쉬운 말로 ("간단해서", "빠르니까", "이 프로젝트 규모에 맞아서")
2. **다른 방법도 있었다** — 최소 1개, 최대 3개
3. **그걸 안 쓴 이유** — 각각 한 줄, 비유 포함

> 포맷:
> - **지금 방식**: [이름] → [왜 골랐는지 쉬운 말로]
> - **다른 방법 1**: [이름] → [왜 안 썼는지 쉬운 말로]
> - **다른 방법 2**: [이름] → [왜 안 썼는지 쉬운 말로]
>
> 예:
> - **지금 방식**: Zustand → 작은 메모장 하나로 충분한 규모라서
> - **다른 방법 1**: Redux → 대형 ERP 시스템용 서류 캐비닛인데, 우리 가게엔 과해요
> - **다른 방법 2**: Context API → 하나 바꾸면 가게 전체가 리셋되는 문제가 있어요

### 4. 더 궁금하면 (Next)

- 관련 파일 1~2개 (경로 제시)
- "이건 뭐야?"로 이어질 수 있는 연관 개념 2~3개를 **쉬운 한 줄 설명**과 함께

> 예:
> - `src/stores/auth-store.ts` — 로그인 상태를 기억해두는 메모장
> - 연관 개념: "토큰"(입장 팔찌), "세션"(로그인 유지 시간), "쿠키"(브라우저가 기억하는 쪽지)

## 말투 규칙

- **항상 "~해요", "~이에요" 체** (반말 금지, 경어체 유지)
- 모든 전문 용어에 괄호 설명 필수: `미들웨어(주방 가기 전 검문소)`
- 한 문단에 전문 용어 **최대 2개** — 넘으면 문단을 나눈다
- 코드 블록은 **5줄 이내**, 핵심 줄에 `// ← 여기가 핵심!` 주석
- "당연히", "아시다시피", "간단히 말해" **금지** — 아무것도 당연하지 않다
- "쉽게 말하면"도 자제 — 애초에 쉽게 말해야 한다
- 이모지 적극 사용 (시각적 구분, 레이어 표시용)

## 컨텍스트 수집 (응답 전 반드시)

1. **대상 파일 Read** — 질문에 해당하는 코드를 실제로 읽는다
2. **git log 확인** — 해당 파일의 최근 커밋 메시지로 "왜" 근거 확보
3. **import/호출 추적** — 이 코드가 어디서 불리고 어디를 부르는지 1단계 확인
4. **프로젝트 구조 파악** — 디렉토리 구조로 음식점 비유 중 어디인지 판단

## 예시

**사용자**: "이 useAuth가 뭐야?"

---

**한 줄 요약**: 이건 "지금 로그인돼 있나?" 확인하는 장치예요.
놀이공원 입장 팔찌 같은 거예요 — 팔찌 차고 있으면 놀이기구 탈 수 있고, 없으면 매표소로 돌아가야 해요.

**전체 그림에서 위치**: **🖥️ 화면(홀)** 쪽이에요.
화면에서 "이 사람 로그인했나?" 확인하고, 안 했으면 로그인 페이지로 보내요.
확인할 때 🚪 통로(API)를 통해 서버에 "이 팔찌 진짜야?" 하고 물어봐요.

**왜 이렇게 만들었나**:
- **지금 방식**: 커스텀 훅(재사용 가능한 확인 장치) + Zustand(작은 메모장)
  → 어디서든 "로그인 확인!" 한 줄이면 끝나서 편해요
- **다른 방법 1**: NextAuth.js(올인원 로그인 패키지)
  → 이 프로젝트는 주방(서버)이 따로 있어서, 홀에서 주방까지 다 해주는 패키지는 오히려 복잡해져요
- **다른 방법 2**: Context API(리액트 기본 공유 기능)
  → 팔찌 상태가 바뀌면 가게 전체가 깜빡이는 문제가 있어요 (리렌더링)

**더 궁금하면**:
- `src/stores/auth-store.ts` — 로그인 메모장 (팔찌 발급 기록)
- `src/api/auth.ts` — 매표소와 대화하는 코드
- 연관 개념: "토큰"(디지털 팔찌), "세션"(팔찌 유효 시간), "라우트 가드"(놀이기구 앞 검표원)
