# 브레인스토밍 — 아이디어를 설계로

아이디어를 1:1 대화를 통해 완성된 설계 문서와 구현 계획으로 만든다.

먼저 프로젝트 컨텍스트를 파악하고, 한 번에 하나씩 질문하며 아이디어를 정제한다. 무엇을 만들지 이해하면 설계를 제시하고 승인을 받는다.

<HARD-GATE>
설계를 제시하고 사용자가 승인하기 전까지 절대 코드를 작성하거나, 구현 스킬을 호출하거나, 스캐폴딩을 하지 않는다. 아무리 간단해 보여도 예외 없다.
</HARD-GATE>

## 안티패턴: "이건 너무 간단해서 설계가 필요 없어"

모든 프로젝트는 이 프로세스를 거친다. TODO 앱, 유틸 함수 하나, 설정 변경 — 전부. "간단한" 프로젝트에서 검증하지 않은 가정이 가장 많은 재작업을 만든다. 정말 간단하면 설계도 짧게(몇 문장) 작성하면 된다. 하지만 반드시 제시하고 승인받아야 한다.

## 체크리스트

반드시 순서대로 완료한다:

1. **프로젝트 컨텍스트 파악** — 파일, 문서, 최근 커밋 확인
2. **명확화 질문** — 한 번에 하나씩. 목적, 제약, 성공 기준 파악
3. **2-3가지 접근법 제안** — 트레이드오프와 추천안 제시
4. **설계 제시** — 복잡도에 맞게 섹션별로. 각 섹션 후 확인
5. **설계 문서 작성** — `docs/specs/YYYY-MM-DD-<topic>-design.md`에 저장
6. **스펙 셀프리뷰** — 플레이스홀더, 모순, 모호함, 스코프 점검
7. **사용자 리뷰** — 스펙 파일 확인 요청, 승인 대기
8. **구현 계획 전환** — 승인 후 구현 계획 작성

## 프로세스

### 아이디어 이해하기

- 현재 프로젝트 상태를 먼저 파악한다 (파일, 문서, 최근 커밋)
- 상세 질문 전에 스코프를 평가한다: 여러 독립 서브시스템을 포함하는 요청이면 즉시 분해를 제안한다
- 프로젝트가 단일 스펙으로 다루기엔 크면 서브 프로젝트로 분해를 돕는다. 각 서브 프로젝트는 자체 스펙 → 계획 → 구현 사이클을 갖는다
- 적절한 스코프면 한 번에 하나씩 질문한다
- 가능하면 객관식, 필요하면 주관식
- **한 메시지에 질문 하나** — 여러 질문이 필요하면 나눈다
- 목적, 제약, 성공 기준에 집중한다

### 접근법 탐색

- 2-3가지 접근법을 트레이드오프와 함께 제안한다
- 추천안을 먼저, 이유와 함께 제시한다
- 대화체로 자연스럽게 설명한다

### 설계 제시

- 무엇을 만들지 이해했다고 판단하면 설계를 제시한다
- 각 섹션은 복잡도에 비례: 단순하면 몇 문장, 복잡하면 200-300단어
- **섹션마다** 괜찮은지 확인한다
- 다루는 내용: 아키텍처, 컴포넌트, 데이터 흐름, 에러 처리, 테스트
- 이해가 안 되면 돌아가서 다시 명확화한다

### 격리와 명확성을 위한 설계

- 시스템을 하나의 목적을 가진 작은 단위로 분해한다
- 각 단위는 명확한 인터페이스로 소통하고, 독립적으로 이해하고 테스트할 수 있어야 한다
- 내부를 읽지 않고도 무엇을 하는지 알 수 있는가? 내부를 바꿔도 소비자가 깨지지 않는가? 아니라면 경계를 다시 잡는다

### 기존 코드베이스에서 작업할 때

- 변경을 제안하기 전에 기존 구조를 탐색한다. 기존 패턴을 따른다
- 현재 작업에 영향을 주는 문제(커진 파일, 불명확한 경계, 엉킨 책임)가 있으면 설계에 개선을 포함한다
- 관련 없는 리팩토링은 제안하지 않는다. 현재 목표에 집중한다

## 설계 이후

### 문서화

- 검증된 설계를 `docs/specs/YYYY-MM-DD-<topic>-design.md`에 저장한다
- 저장 후 커밋한다

### 스펙 셀프리뷰

문서 작성 후 새로운 눈으로 점검한다:

1. **플레이스홀더 스캔:** "TBD", "TODO", 미완성 섹션, 모호한 요구사항이 있는가? 수정한다.
2. **내부 일관성:** 섹션 간 모순이 있는가? 아키텍처와 기능 설명이 일치하는가?
3. **스코프 점검:** 단일 구현 계획으로 충분한 규모인가, 분해가 필요한가?
4. **모호함 점검:** 두 가지로 해석 가능한 요구사항이 있는가? 하나를 골라 명시한다.

문제를 발견하면 인라인으로 수정한다. 재리뷰 불필요 — 수정하고 넘어간다.

### 사용자 리뷰 게이트

셀프리뷰 통과 후 사용자에게 확인을 요청한다:

> "스펙을 `<경로>`에 작성했습니다. 구현 계획으로 넘어가기 전에 확인해 주세요. 수정이 필요하면 말씀해 주세요."

사용자 응답을 기다린다. 변경 요청이 오면 수정하고 셀프리뷰를 다시 돌린다. 승인이 오면 다음으로 진행한다.

### 구현 계획

- 승인된 스펙을 기반으로 구현 계획을 작성한다
- 단계별로 무엇을 어떤 순서로 구현할지, 각 단계의 검증 기준은 무엇인지 명시한다
- 가능하면 tests.json 태스크로 분해한다

## 비주얼 컴패니언

시각적 질문(레이아웃, 목업, 다이어그램)이 예상될 때 로컬 브라우저를 열어 인터랙티브하게 진행할 수 있다.

**제안 타이밍:** 시각적 콘텐츠가 필요한 질문이 올 것 같을 때 한 번만 제안한다 (다른 내용과 섞지 말고 별도 메시지로):

> "진행하다 보면 레이아웃이나 구조를 브라우저에서 바로 보여드릴 수 있어요. 로컬 서버를 띄워서 목업이나 다이어그램을 시각적으로 보여드리는 기능인데, 써볼까요? (로컬 URL을 열어야 합니다)"

수락하면 `skills/brainstorm/visual-companion.md`를 읽고 안내대로 진행한다.

**질문마다 판단:** 수락했더라도 매 질문마다 "보는 게 이해에 도움이 되는가?"를 판단한다.
- 브라우저 사용: UI 목업, 레이아웃 비교, 아키텍처 다이어그램, 디자인 옵션
- 터미널 사용: 요구사항 질문, 개념 선택, 트레이드오프 비교, 기술 결정

## 핵심 원칙

- **한 번에 질문 하나** — 여러 질문으로 압도하지 않는다
- **객관식 선호** — 가능하면 주관식보다 쉽게 답할 수 있게
- **YAGNI 철저히** — 불필요한 기능은 설계에서 제거한다
- **대안 탐색** — 정하기 전에 항상 2-3가지 접근법을 제안한다
- **점진적 검증** — 설계를 제시하고, 승인받고, 다음으로
- **유연하게** — 이해가 안 되면 돌아가서 명확화한다
- **한국어 우선** — 한국어로 대화하되, 기술 용어는 영어 병행 가능
