# 체계적 디버깅

## 개요

무작위 수정은 시간 낭비이며 새로운 버그를 만든다. 빠른 패치는 근본 문제를 감춘다.

**핵심 원칙:** 수정을 시도하기 전에 반드시 근본 원인을 찾는다. 증상만 고치는 것은 실패다.

**이 프로세스의 문자를 위반하는 것은 디버깅의 정신을 위반하는 것이다.**

## 철칙

```
근본 원인 조사 없이 수정 금지
```

Phase 1을 완료하지 않았다면, 수정을 제안할 수 없다.

## 언제 사용하나

모든 기술적 이슈에 사용:
- 테스트 실패
- 프로덕션 버그
- 예상치 못한 동작
- 성능 문제
- 빌드 실패
- 통합 이슈

**특히 다음 상황에서 반드시 사용:**
- 시간 압박이 있을 때 (긴급 상황은 추측을 유혹한다)
- "빠른 수정 하나"가 뻔해 보일 때
- 이미 여러 수정을 시도했을 때
- 이전 수정이 작동하지 않았을 때
- 이슈를 완전히 이해하지 못할 때

**건너뛰지 마라:**
- 이슈가 단순해 보여도 (단순한 버그에도 근본 원인이 있다)
- 바쁠 때 (서두르면 재작업이 보장된다)
- 지금 당장 고쳐야 할 때 (체계적 접근이 삽질보다 빠르다)

## 4단계 프로세스

각 단계를 완료한 후에만 다음 단계로 진행할 수 있다.

### Phase 1: 근본 원인 조사

**수정을 시도하기 전에 반드시:**

1. **에러 메시지를 주의 깊게 읽기**
   - 에러나 경고를 건너뛰지 말 것
   - 정확한 해결책이 포함되어 있는 경우가 많다
   - 스택 트레이스를 끝까지 읽기
   - 라인 번호, 파일 경로, 에러 코드 기록

2. **일관되게 재현하기**
   - 안정적으로 트리거할 수 있나?
   - 정확한 단계는?
   - 매번 발생하나?
   - 재현 불가 -> 데이터를 더 수집, 추측 금지

3. **최근 변경 확인**
   - 이 문제를 유발할 수 있는 변경은?
   - git diff, 최근 커밋
   - 새 의존성, 설정 변경
   - 환경 차이

4. **다중 컴포넌트 시스템에서 증거 수집**

   **시스템에 여러 컴포넌트가 있을 때 (CI -> 빌드 -> 서명, API -> 서비스 -> DB):**

   **수정 제안 전에 진단 계측 추가:**
   ```
   각 컴포넌트 경계마다:
     - 컴포넌트에 들어오는 데이터 로깅
     - 컴포넌트에서 나가는 데이터 로깅
     - 환경/설정 전파 검증
     - 각 레이어의 상태 확인

   한 번 실행하여 어디서 깨지는지 증거 수집
   그 다음 증거를 분석하여 실패 컴포넌트 식별
   그 다음 해당 컴포넌트를 집중 조사
   ```

   **예시 (다중 레이어 시스템):**
   ```bash
   # 레이어 1: 워크플로우
   echo "=== 워크플로우에서 사용 가능한 시크릿: ==="
   echo "IDENTITY: ${IDENTITY:+SET}${IDENTITY:-UNSET}"

   # 레이어 2: 빌드 스크립트
   echo "=== 빌드 스크립트의 환경 변수: ==="
   env | grep IDENTITY || echo "IDENTITY가 환경에 없음"

   # 레이어 3: 서명 스크립트
   echo "=== 키체인 상태: ==="
   security list-keychains
   security find-identity -v

   # 레이어 4: 실제 서명
   codesign --sign "$IDENTITY" --verbose=4 "$APP"
   ```

   **이것이 보여주는 것:** 어떤 레이어가 실패하는지 (시크릿 -> 워크플로우 O, 워크플로우 -> 빌드 X)

5. **데이터 흐름 추적**

   **에러가 콜 스택 깊은 곳에 있을 때:**

   - 잘못된 값이 어디서 시작되나?
   - 무엇이 잘못된 값으로 이것을 호출했나?
   - 소스를 찾을 때까지 위로 계속 추적
   - 증상이 아닌 소스에서 수정

### Phase 2: 패턴 분석

**수정 전에 패턴을 찾기:**

1. **작동하는 예시 찾기**
   - 같은 코드베이스에서 유사하게 작동하는 코드 찾기
   - 깨진 것과 비슷한데 작동하는 것은?

2. **레퍼런스와 비교**
   - 패턴을 구현 중이라면, 참조 구현을 완전히 읽기
   - 훑어보지 말고 모든 줄 읽기
   - 적용하기 전에 패턴을 완전히 이해

3. **차이점 식별**
   - 작동하는 것과 깨진 것의 차이는?
   - 아무리 작아도 모든 차이를 나열
   - "그건 상관없을 거야"라고 가정하지 않기

4. **의존성 이해**
   - 이것이 필요로 하는 다른 컴포넌트는?
   - 어떤 설정, 구성, 환경?
   - 어떤 가정을 하고 있나?

### Phase 3: 가설과 검증

**과학적 방법:**

1. **단일 가설 수립**
   - 명확하게 기술: "Y 때문에 X가 근본 원인이라고 생각한다"
   - 기록
   - 모호하지 않게, 구체적으로

2. **최소한으로 테스트**
   - 가설을 검증할 수 있는 가장 작은 변경
   - 한 번에 하나의 변수만
   - 여러 가지를 한꺼번에 고치지 않기

3. **계속하기 전에 확인**
   - 작동했나? -> Phase 4로
   - 작동 안 했나? -> 새로운 가설 수립
   - 기존 수정 위에 추가 수정을 쌓지 않기

4. **모를 때**
   - "X를 이해하지 못한다"고 말하기
   - 아는 척하지 않기
   - 도움 요청
   - 더 조사

### Phase 4: 구현

**증상이 아닌 근본 원인을 수정:**

1. **실패하는 테스트 케이스 작성**
   - 가장 간단한 재현
   - 가능하면 자동화 테스트
   - 프레임워크가 없으면 일회용 테스트 스크립트
   - 수정 전에 반드시 있어야 함
   - 적절한 실패 테스트 작성에는 `wj:tdd` 스킬 사용

2. **단일 수정 구현**
   - 식별된 근본 원인을 해결
   - 한 번에 하나의 변경만
   - "김에" 개선 금지
   - 번들 리팩토링 금지

3. **수정 확인**
   - 테스트가 통과하나?
   - 다른 테스트가 깨지지 않았나?
   - 이슈가 실제로 해결되었나?

4. **수정이 작동하지 않으면**
   - 중단
   - 시도한 수정 횟수 세기
   - 3회 미만: Phase 1로 돌아가서 새로운 정보로 재분석
   - **3회 이상: 중단하고 아키텍처에 의문을 제기 (아래 5번)**
   - 아키텍처 논의 없이 4번째 수정을 시도하지 않기

5. **3회 이상 수정 실패: 아키텍처 의문 제기**

   **아키텍처 문제를 나타내는 패턴:**
   - 각 수정이 다른 곳의 새로운 공유 상태/결합/문제를 드러냄
   - 수정이 "대규모 리팩토링"을 필요로 함
   - 각 수정이 다른 곳에서 새로운 증상을 만듦

   **중단하고 근본을 의문시:**
   - 이 패턴이 근본적으로 건전한가?
   - "관성으로 계속 밀고 가는 것"은 아닌가?
   - 증상 수정을 계속하는 것 vs 아키텍처를 리팩토링해야 하는가?

   **더 많은 수정을 시도하기 전에 사용자와 논의**

   이것은 실패한 가설이 아니다 -- 잘못된 아키텍처다.

## 위험 신호 - 중단하고 프로세스를 따르라

이런 생각이 드는 순간:
- "일단 빠르게 고치고 나중에 조사하자"
- "X를 바꿔보고 되는지 보자"
- "여러 변경을 추가하고 테스트를 돌리자"
- "테스트는 건너뛰고 수동으로 확인하자"
- "아마 X일 거야, 그걸 고치자"
- "완전히 이해하지 못하지만 이게 작동할 수도"
- "패턴은 X라는데 나는 다르게 적용하겠다"
- "주요 문제들은 이거다: [조사 없이 수정 목록 나열]"
- 데이터 흐름 추적 전에 해결책 제안
- **"한 번만 더 수정 시도" (이미 2회 이상 시도 후)**
- **각 수정이 다른 곳에서 새로운 문제를 드러냄**

**이 모두가 의미하는 것: 중단. Phase 1로 돌아가라.**

**3회 이상 수정 실패:** 아키텍처에 의문을 제기하라 (Phase 4.5)

## 사용자가 잘못하고 있다는 신호

**다음 피드백에 주의:**
- "그거 안 되는 거 아니야?" - 검증 없이 가정했음
- "그러면 보여줄 수 있어...?" - 증거 수집을 했어야 함
- "추측 그만해" - 이해 없이 수정을 제안하고 있음
- "제대로 생각해봐" - 증상만 아니라 근본을 따져야 함
- "우리 막힌 거야?" (좌절) - 접근 방식이 작동하지 않음

**이것을 보면:** 중단. Phase 1로 돌아가라.

## 흔한 합리화

| 핑계 | 현실 |
|------|------|
| "이슈가 단순해서 프로세스 필요 없어" | 단순한 이슈에도 근본 원인이 있다. 단순한 버그에 프로세스는 빠르다. |
| "긴급해서 프로세스 할 시간 없어" | 체계적 디버깅이 추측-확인 삽질보다 빠르다. |
| "일단 이것부터 해보고 조사하자" | 첫 번째 수정이 패턴을 만든다. 처음부터 제대로 하라. |
| "수정 확인 후에 테스트 작성할게" | 테스트 없는 수정은 지속되지 않는다. 테스트 먼저가 증명한다. |
| "한꺼번에 여러 수정하면 시간 절약" | 무엇이 작동했는지 분리 불가. 새 버그를 유발한다. |
| "레퍼런스가 너무 길어, 패턴을 적용하겠다" | 부분적 이해는 버그를 보장한다. 완전히 읽어라. |
| "문제를 봤어, 고치겠다" | 증상을 보는 것 != 근본 원인을 이해하는 것. |
| "한 번만 더 수정 시도" (2회 이상 실패 후) | 3회 이상 실패 = 아키텍처 문제. 패턴을 의문시하라, 다시 고치지 마라. |

## 빠른 참조

| 단계 | 주요 활동 | 성공 기준 |
|------|----------|----------|
| **1. 근본 원인** | 에러 읽기, 재현, 변경 확인, 증거 수집 | 무엇이 그리고 왜를 이해 |
| **2. 패턴** | 작동하는 예시 찾기, 비교 | 차이점 식별 |
| **3. 가설** | 이론 수립, 최소한 테스트 | 확인 또는 새 가설 |
| **4. 구현** | 테스트 작성, 수정, 검증 | 버그 해결, 테스트 통과 |

## 프로세스가 "근본 원인 없음"을 보여줄 때

체계적 조사가 이슈가 진정 환경, 타이밍, 또는 외부 요인에 의한 것임을 밝혀낸 경우:

1. 프로세스를 완료한 것이다
2. 조사한 내용을 문서화
3. 적절한 처리 구현 (재시도, 타임아웃, 에러 메시지)
4. 향후 조사를 위한 모니터링/로깅 추가

**하지만:** "근본 원인 없음"의 95%는 불완전한 조사다.

## 관련 스킬

- **wj:tdd** - 실패하는 테스트 케이스 작성 (Phase 4, Step 1)
- **wj:verify** - 수정이 작동했는지 완료 주장 전에 확인

## 실전 효과

디버깅 세션 기록:
- 체계적 접근: 15-30분에 수정
- 무작위 수정 접근: 2-3시간 삽질
- 첫 번째 수정 성공률: 95% vs 40%
- 새 버그 도입: 거의 제로 vs 빈번
