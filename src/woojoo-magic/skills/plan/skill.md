# 구현 계획 작성

## 개요

코드베이스에 대한 맥락이 전혀 없는 엔지니어를 위한 포괄적인 구현 계획을 작성한다. 각 태스크에서 어떤 파일을 건드려야 하는지, 코드, 테스트, 확인해야 할 문서, 테스트 방법까지 모든 것을 문서화한다. 전체 계획을 소화 가능한 태스크로 제공. DRY. YAGNI. TDD. 자주 커밋.

숙련된 개발자지만 우리 도구나 문제 도메인에 대해 거의 모르는 사람을 가정한다. 좋은 테스트 설계에 익숙하지 않다고 가정한다.

**시작 시 안내:** "wj:plan 스킬을 사용하여 구현 계획을 작성합니다."

**계획 저장 위치:** `docs/specs/plans/YYYY-MM-DD-<feature-name>.md`
- (사용자의 저장 위치 선호가 이 기본값보다 우선한다)

## 범위 확인

스펙이 여러 독립 서브시스템을 다루는 경우, 브레인스토밍 단계에서 하위 프로젝트 스펙으로 분리했어야 한다. 그렇지 않았다면, 별도 계획으로 나누는 것을 제안하라 -- 서브시스템당 하나. 각 계획은 자체적으로 작동하고 테스트 가능한 소프트웨어를 산출해야 한다.

## 파일 구조

태스크를 정의하기 전에, 어떤 파일이 생성 또는 수정될지와 각 파일의 책임을 매핑한다. 여기서 분해 결정이 확정된다.

- 명확한 경계와 잘 정의된 인터페이스로 단위를 설계. 각 파일은 하나의 명확한 책임을 가져야 한다.
- 한 번에 맥락에 담을 수 있는 코드에 대해 가장 잘 추론하며, 파일이 집중되어 있을 때 편집이 더 안정적이다. 너무 많은 일을 하는 큰 파일보다 작고 집중된 파일을 선호.
- 함께 변경되는 파일은 함께 위치해야 한다. 기술 레이어가 아닌 책임으로 분리.
- 기존 코드베이스에서는 확립된 패턴을 따른다. 큰 파일을 사용하는 코드베이스라면 일방적으로 구조를 바꾸지 않되, 수정 중인 파일이 다루기 어려울 정도로 커졌다면 계획에 분리를 포함하는 것이 합리적이다.

이 구조가 태스크 분해를 결정한다. 각 태스크는 독립적으로 의미 있는 자체 완결적 변경을 산출해야 한다.

## 한 입 크기 태스크 단위

**각 단계는 하나의 액션 (2-5분):**
- "실패하는 테스트 작성" - 단계
- "실패하는지 실행하여 확인" - 단계
- "테스트를 통과하는 최소한의 코드 구현" - 단계
- "테스트를 실행하여 통과 확인" - 단계
- "커밋" - 단계

## 계획 문서 헤더

**모든 계획은 반드시 이 헤더로 시작:**

```markdown
# [기능명] 구현 계획

> **에이전트 작업자용:** wj:team 스킬 또는 직접 실행으로 이 계획을 태스크별로 구현하세요. 단계는 체크박스 (`- [ ]`) 문법으로 추적합니다.

**목표:** [이것이 무엇을 만드는지 한 문장으로]

**아키텍처:** [접근 방식 2-3문장]

**기술 스택:** [핵심 기술/라이브러리]

---
```

## 태스크 구조

````markdown
### Task N: [컴포넌트명]

**파일:**
- 생성: `exact/path/to/file.py`
- 수정: `exact/path/to/existing.py:123-145`
- 테스트: `tests/exact/path/to/test.py`

- [ ] **Step 1: 실패하는 테스트 작성**

```python
def test_specific_behavior():
    result = function(input)
    assert result == expected
```

- [ ] **Step 2: 테스트 실행하여 실패 확인**

실행: `pytest tests/path/test.py::test_name -v`
예상: FAIL - "function not defined"

- [ ] **Step 3: 최소한의 구현 작성**

```python
def function(input):
    return expected
```

- [ ] **Step 4: 테스트 실행하여 통과 확인**

실행: `pytest tests/path/test.py::test_name -v`
예상: PASS

- [ ] **Step 5: 커밋**

```bash
git add tests/path/test.py src/path/file.py
git commit -m "feat: 특정 기능 추가"
```
````

## 플레이스홀더 금지

모든 단계는 엔지니어가 필요로 하는 실제 내용을 담아야 한다. 다음은 **계획 실패**다 -- 절대 쓰지 마라:
- "TBD", "TODO", "나중에 구현", "세부사항 채우기"
- "적절한 에러 핸들링 추가" / "유효성 검사 추가" / "엣지 케이스 처리"
- "위 코드에 대한 테스트 작성" (실제 테스트 코드 없이)
- "Task N과 유사" (코드를 반복하라 -- 엔지니어는 순서 없이 읽을 수 있다)
- 코드 블록 없이 무엇을 해야 하는지만 설명하는 단계 (코드 단계에는 코드 블록 필수)
- 어떤 태스크에도 정의되지 않은 타입, 함수, 메서드 참조

## 기억할 것
- 항상 정확한 파일 경로
- 모든 단계에 완전한 코드 -- 코드를 변경하는 단계라면 코드를 보여줄 것
- 예상 출력과 함께 정확한 명령어
- DRY, YAGNI, TDD, 자주 커밋

## 자체 리뷰

완전한 계획을 작성한 후, 스펙을 새로운 눈으로 보고 계획과 대조 검사한다. 이것은 직접 수행하는 체크리스트다 -- 서브에이전트 디스패치가 아니다.

**1. 스펙 커버리지:** 스펙의 각 섹션/요구사항을 훑어본다. 그것을 구현하는 태스크를 가리킬 수 있나? 누락 사항을 나열.

**2. 플레이스홀더 스캔:** 계획에서 위험 신호를 검색 -- "플레이스홀더 금지" 섹션의 패턴. 수정.

**3. 타입 일관성:** 이후 태스크에서 사용한 타입, 메서드 시그니처, 프로퍼티 이름이 이전 태스크에서 정의한 것과 일치하나? Task 3에서 `clearLayers()`인데 Task 7에서 `clearFullLayers()`면 버그다.

문제를 발견하면 인라인으로 수정. 재리뷰는 불필요 -- 그냥 수정하고 넘어간다. 태스크가 없는 스펙 요구사항을 발견하면 태스크를 추가한다.

## 실행 핸드오프

계획을 저장한 후, 실행 방식을 제안:

**"계획 완료. `docs/specs/plans/<filename>.md`에 저장했습니다. 두 가지 실행 옵션:**

**1. 팀 기반 (권장)** - wj:team 스킬로 태스크별 에이전트 디스패치, 태스크 간 리뷰, 빠른 반복

**2. 직접 실행** - 이 세션에서 직접 태스크를 실행, 체크포인트로 배치 실행

**어떤 방식으로 할까요?"**

**팀 기반 선택 시:**
- wj:team 스킬을 사용하여 태스크별 에이전트 디스패치 + 리뷰

**직접 실행 선택 시:**
- 이 세션에서 체크포인트를 두고 배치 실행
